# 纯按钮触发重构方案：彻底解决后台监控导致的性能问题

## 一、动机

当前插件存在严重的性能问题：

- `logseq.DB.onChanged` 全局监听器永久注册，任何键盘输入都触发全页扫描
- `logseq.DB.onBlockChanged` 监听器每次变更时累积，从不释放
- 500ms 间隔的 `scanAndInjectIndicators` DOM 轮询
- 级联循环：编号 → `Editor.updateBlock` → 又触发 `onChanged` → 再次编号
- 暂停功能只停了"是否执行编号"的判断，**没有停止上述任何监听器**

**根因追溯**：所有问题都源于"后台实时监控"的设计理念。改为纯按钮触发可以从根本上消除这些持续性资源占用。

## 二、新旧架构对比

### 改动前：后台监控型

```
                  ┌─ 启动时 ─→ onChanged 注册 ──────────────────┐
                  │                                              ▼
用户 → 编辑笔记 ──→ onChanged 触发 ─→ 检测 heading 属性 ─→ updateToc()
                                          │              │
                                          ▼              ▼
                                    onBlockChanged  refreshPageHeaders
                                    泄漏累积（从不     │
                                    清理）            ├─ applyHeadingNumbersToPage
                                                      │    └─ 多处重复调用
                                                      ├─ onBlockChanged 注册
                                                      │
                    ┌─ 500ms 轮询 ─→ scanAndInjectIndicators
                    │                 (querySelectorAll + getBoundingClientRect)
                    │
暂停按钮 → 仅设 pausedPages 标记
           ❌ 不清理已累积的监听器
           ❌ 不停止 onChanged
```

### 改动后：纯按钮触发型

```
重新编号 [按钮] ─→ applyHeadingNumbersToPage() ─→ 完成 ✓
                      │
                      └→ normalizePageHeadingsInternal() (如果启用)

打开笔记 ──→ onRouteChanged ─→ routeCheck() ─→ refreshPageHeaders()
                                                   ← 自动更新 TOC
刷新TOC [按钮] ─→ routeCheck() ─→ refreshPageHeaders()
                                     ← 手动补充

后台没有任何定时器、DB.onChanged、DOM 轮询
唯一保留的监听器: onRouteChanged（离散事件，页面切换时触发一次）
```

## 三、设置项简化

### 移除的配置项

| 设置键 | 原用途 | 移除原因 |
|--------|--------|---------|
| `headingNumberFileEnable` | 选择「全局自动编号 / 单页面手动开关 / 关闭自动编号」 | 不再需要模式选择，按钮总是手动触发 |

### 保留的配置项

| 设置键 | 保留原因 |
|--------|---------|
| `autoHeadingLevelEnabled` | 在点击编号按钮时仍可生效（由 `applyHeadingNumbersToPage` 内部调用） |
| `autoHeadingLevelReserveH1` | 同上，属于 autoHeadingLevelEnabled 的子项 |

### 白名单系统（移除）

原白名单（`pageWhitelist.ts`）中仅保留**块状态系统**（skip/lock/repeat），页面级白名单不再需要。

## 五、行为矩阵

| 场景 | 改动前 | 改动后 |
|------|--------|--------|
| 编辑笔记内容 | 触发 onChanged → 扫描全页 + TOC 刷新 | 不做任何事 |
| 添加新标题 | 触发 onChanged → 编号 + TOC 自动更新 | 不做任何事（不需要编号，如有需要点「重新编号」按钮） |
| 切换页面 | onRouteChanged → routeCheck → 全量处理 | onRouteChanged → routeCheck → TOC 刷新 ✓ |
| 点击「重新编号」 | 触发编号 + 暂停相关逻辑 | 直接编号当前页 |
| 点击「刷新 TOC」 | routeCheck → refreshPageHeaders | 同左（保持不变） |
| 右键→跳过/锁定/重复 | 修改状态后自动重新编号 | 修改状态 → 提示用户点击重新编号按钮 |
| 插件启动 | 注册 onChanged + 启动 500ms 轮询 | 只渲染 TOC 容器 + 单次 routeCheck |

## 六、副作用与注意事项

### 1. 侧边栏 TOC 不再实时更新
用户在笔记中添加/删除标题后，TOC 不会自动刷新。用户需点击 TOC 面板右上角的刷新按钮（已有该按钮）。

### 2. `lse-heading-enabled` CSS 类
不再需要根据白名单动态开关，相关逻辑可以简化为始终添加。

### 3. 右键菜单交互变化
点击「跳过/锁定/重复」后，不再自动触发全页重编号。用户需要手动点击工具栏的「重新编号」按钮来应用更改。建议在消息提示中明确引导。

### 4. 向后兼容
- 已有编号的笔记不受影响（编号文本已写入文件）
- 现有配置的 `headingNumberFileEnable` 值会被新版本忽略
- 沙箱存储中的白名单条目会被保留（但不再读取），不影响块状态
